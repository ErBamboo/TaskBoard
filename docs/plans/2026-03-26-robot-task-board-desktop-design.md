# 机器人竞赛任务面板 Windows 桌面版设计说明

## 当前背景

当前仓库已经有一套可运行的 Web MVP：

- 独立账号登录
- 我的任务首页
- 项目列表与项目看板
- 任务创建、编辑、状态流转
- 管理员维护成员、项目、子系统、里程碑

但这套形态仍然偏“可部署的网页系统”，距离目标中的“极致易用性”还有明显差距。用户仍然需要理解网址、浏览器安装入口、网页心智和后台初始化心智，不符合“像普通 Windows 软件一样安装、打开、直接开始用”的要求。

因此，本设计在保留现有业务模型的前提下，正式将产品交付形态调整为：

- 管理员统一部署一次云端服务
- 团队成员只下载安装 Windows 客户端
- 安装后桌面出现图标，双击直接打开
- 登录后默认记住状态，下次直接进入系统

本设计用于补充并部分取代 `2026-03-25-robot-task-board-design.md` 中“网页优先交付”的产品形态假设。

## 1. 目标

为机器人竞赛团队提供一套真正面向最终用户的桌面软件交付形态，使成员可以用接近普通 Windows 应用的方式使用任务系统。

首要目标是：

- 一键安装
- 一键打开
- 只登录一次
- 之后直接进入“我的任务”
- 管理员首次使用时，通过向导完成初始化

## 2. 已确认决策

- 用户设备默认使用 Windows
- 管理员可以统一部署一次云端服务
- 成员不需要自行配置环境变量、数据库、服务地址
- 当前业务模型继续保留：
  - 项目
  - 子系统
  - 里程碑
  - 任务
  - 任务关联
- 账号模型继续保留：
  - 每个成员独立账号
  - 唯一管理员账号
- 交付形态调整为：
  - `云端服务 + Windows 桌面客户端`
- 技术方案优先考虑：
  - `Tauri`

## 3. 方案对比与结论

### 3.1 继续 PWA

优点：

- 基于现有代码改动最小
- 不需要引入 Rust/Tauri

问题：

- 安装方式仍然依赖浏览器
- 用户对“添加到主屏幕/安装应用”的理解成本高
- 桌面图标、自动更新、首次启动体验都不稳定
- 仍然保留明显网页心智

结论：

不满足“极致易用性”目标。

### 3.2 Electron 桌面客户端

优点：

- 生态成熟
- 文档多
- 与 Web 技术栈衔接直接

问题：

- 安装包和运行体积更大
- 资源占用更高
- 对 6-8 人学生团队来说性价比不够好

结论：

可行，但不是首选。

### 3.3 Tauri 桌面客户端

优点：

- Windows 安装包形态更接近普通软件
- 包体更小
- 运行更轻
- 适合在保留现有前端的同时补足桌面交付体验

问题：

- 需要增加桌面端工程层
- 需要重新设计首次启动与安装体验

结论：

这是推荐方案。

## 4. 总体架构

系统拆分为两部分：

- 云端服务
- Windows 桌面客户端

### 4.1 云端服务职责

云端服务继续承载真实业务数据和业务规则，仍然是系统唯一事实来源。

负责内容包括：

- 认证与会话
- 成员、项目、子系统、里程碑、任务数据
- 权限判断
- 任务创建、编辑、状态流转
- 首次初始化向导的数据写入

云端服务不因为桌面客户端的引入而拆成第二套业务逻辑。

### 4.2 Windows 客户端职责

Windows 客户端只负责把系统变成“像软件一样被使用”的形态。

负责内容包括：

- 安装包
- 桌面图标
- 双击启动
- 固定服务入口
- 记住登录状态
- 自动更新
- 网络异常与版本更新提示

Windows 客户端不保存核心业务事实，也不独立维护任务规则。

## 5. 产品原则

- 云端只保留一份真实业务数据
- 客户端只增强交付体验，不复制业务系统
- 初始化流程模板化，减少管理员第一次使用的配置成本
- 普通成员入口极简，不暴露系统结构复杂度
- 继续沿用“按交付对象组织，而不是按技术组组织”的核心业务模型

## 6. 业务模型保持不变

桌面化不改变核心业务结构，仍然是：

- 项目
- 子系统
- 里程碑
- 任务
- 任务关联

依然遵循：

- 一个任务只保留一个主负责人
- 跨组协作拆成多条任务
- 用联调任务表达集成阶段
- 技术组作为标签，不作为主层级

## 7. 主要用户流程

### 7.1 普通成员流程

1. 下载安装包
2. 双击安装
3. 安装完成后桌面出现图标
4. 双击图标打开软件
5. 首次登录账号
6. 登录成功后直接进入“我的任务”
7. 后续再次打开时默认保留登录状态，直接进入系统

### 7.2 管理员流程

1. 下载并安装客户端
2. 打开软件
3. 登录管理员账号
4. 如果系统尚未初始化，自动进入首次启动向导
5. 完成向导后进入管理台或“我的任务”

## 8. 首次启动向导

首次启动向导只在管理员第一次进入空系统时出现，用来替代当前偏后台式的手工初始化。

### 8.1 步骤一：战队基础信息

字段建议：

- 战队名称
- 当前赛季名称

目标：

- 形成系统基础识别信息
- 让产品从第一步就更像真正的软件，而不是后台管理台

### 8.2 步骤二：初始项目

允许管理员快速创建 1 到 3 个项目。

每个项目只要求：

- 项目名称
- 项目说明（可选）

典型示例：

- 英雄机器人
- 步兵机器人
- 视觉测试平台

### 8.3 步骤三：默认子系统模板

每个项目默认自动生成一组“交付对象导向”的子系统，而不是技术组名称。

建议模板：

- 底盘
- 云台
- 发射机构
- 视觉
- 通信
- 电源
- 上位机

这样既保留了项目结构的一致性，也避免管理员第一次使用时逐条手工录入。

### 8.4 步骤四：成员导入

每位成员只输入：

- 姓名
- 邮箱

角色默认是普通成员。

限制：

- 不开放多个管理员
- 管理员账号唯一

### 8.5 步骤五：完成初始化

系统自动生成：

- 项目
- 默认子系统
- 基础成员账号

完成后提供两个明确入口：

- 进入管理控制台
- 进入我的任务

## 9. 普通成员界面原则

普通成员的最终体验必须尽量收缩成少量固定动作：

- 看“我的任务”
- 进入“项目”
- 创建任务
- 更新任务状态

因此，普通成员顶部导航只保留：

- 我的任务
- 项目

不会展示：

- 初始化配置
- 服务连接配置
- 系统部署概念
- 多余管理结构

## 10. 桌面客户端能力边界

### 10.1 应该包含

- Windows 安装包
- 桌面快捷方式
- 应用图标
- 默认记住登录状态
- 固定服务地址
- 版本更新检查
- 自动更新
- 网络异常提示

### 10.2 不应包含

- 本地业务数据库
- 本地离线任务写入
- 第二套权限逻辑
- 第二套项目管理逻辑
- 本地独立账户系统

原因是这些能力会制造第二份事实来源，增加维护成本和数据一致性风险。

## 11. 服务地址与环境策略

为了实现“成员只安装客户端就能用”，服务地址必须由管理员统一部署，不让普通成员手动配置。

推荐做法：

- 开发版客户端指向开发环境
- 正式版客户端指向生产环境
- 服务地址在构建时固化进桌面客户端配置

成员端不提供可见的服务地址编辑入口。

## 12. 登录与会话策略

- 首次登录使用邮箱与密码
- 登录成功后持久化会话
- 再次打开客户端时自动恢复
- 主动退出登录后清除本地凭证

这套策略的目标是：

- 第一次登录后尽量不再打断成员日常使用

## 13. 更新策略

桌面客户端需要提供基础自动更新能力。

目标效果：

- 成员不需要手动重新下载安装包
- 管理员发布新版本后，客户端可以提示更新或自动安装

范围控制：

- 首版只做基础更新检查与应用
- 不做复杂分阶段灰度发布

## 14. 网络与错误处理

系统仍然是在线协作产品，因此网络断开时必须明确提示，而不是伪造“可正常使用”的状态。

建议行为：

- 网络异常时提示“当前无法连接任务中枢”
- 不允许离线创建或编辑任务
- 提供重新连接和重试入口

## 15. 安全与运维边界

对于学生竞赛团队，安全与运维策略保持“足够稳妥，但不过度企业化”：

- 管理员统一维护生产服务
- 客户端不暴露服务角色密钥
- 普通成员只使用邮箱密码登录
- 不做复杂审计和审批
- 不做多管理员

## 16. 成功标准

- 普通成员完成安装后，30 秒内能进入系统首页
- 第二次打开软件无需重新登录
- 管理员首次进入时，5 分钟内可完成基础初始化
- 团队成员不需要了解任何部署、环境变量和数据库概念
- 软件整体使用心智更接近普通 Windows 应用，而不是网页后台

## 17. 与现有仓库的关系

当前仓库中的 Next.js Web 应用不是被废弃，而是被重新定位为：

- 云端产品前端主体
- 桌面客户端中承载的业务界面来源

下一阶段工作重点不再是继续堆 Web 页面的基础功能，而是：

- 增加 Tauri 桌面壳
- 增加首次启动向导
- 增加安装、更新、登录保持等桌面能力

## 18. 后续扩展方向

待桌面版稳定后，再考虑：

- 开机自动启动
- 系统托盘入口
- 桌面消息提醒
- 更完整的成员导入方式
- 批量初始化项目模板
